## 動機

現状の円柱アリーナは `ArenaRenderer` の **絵だけ** で物理的な閉じ込めが無い ([constants.ts:613](../src/components/game/constants.ts:613) のコメント "world-frame static cylinder, visual guide only")。テストプレイヤーが境界を超えてどんどん遠くに飛んでいき、攻防が成立しない。

→ 周期的境界条件 (PBC) のトーラス topology に切替。「-L まで行ったら本当に +L に飛ぶ」物理を導入。
→ 現状の円柱モードはオプションとしてコード保持 (boundaryMode で切替、UI 撤去 + URL hash override)。

## 設計の核

**state の source of truth は unwrapped 連続値**、wrap は **描画変換** と **距離計算 helper** に閉じ込める。

理由: state を本当に wrap させると snapshot serialization / worldLine ring buffer の連続性 / 過去光円錐の covering space 議論 / segment offset 管理 / network 受信時の wrap 検出が芋づるで複雑化する。state を unwrapped に保ち wrap を「物理が見る空間 = 距離計算の最短画像」と「描画が見る空間 = 観測者中心の primary cell」の 2 箇所に閉じ込めると、必要な改修が局所化する。

## 3 つのヘルパで全体をカバー

```ts
// physics/torus.ts (新規)

// 最短画像 delta: |Δ| > L なら 2L 加減算で primary cell 内に
function minImageDelta(d: number, L: number): number;

// 観測者の primary cell の image cell 番号 (= worldLine の wrap 跨ぎ判定)
function imageCell(p: {x: number, y: number}, obs: {x: number, y: number}, L: number): {kx: number, ky: number};

// observer 中心 wrap: 観測者 [obs-L, obs+L]² 内に折り畳む
function displayPos(p: {x, y}, obs: {x, y}, L: number): {x, y};
//   = obs + minImageDelta(p - obs, L)
//   ただし observer rest frame に持ち込む既存の display 変換と統合する場合は
//   delta だけ返す (= minImageDelta vector) もあり。実装時に既存 transformEventForDisplay
//   と整合させる
```

worldLine の wrap 跨ぎ判定:

```ts
function isWrapCrossing(p0, p1, obs, L) {
  // 判定 1: 生 unwrapped delta が L 超え (broadcast 欠落 / frame 落ち / 高速移動の defensive 検出)
  if (Math.abs(p1.x - p0.x) > L) return true;
  if (Math.abs(p1.y - p0.y) > L) return true;
  // 判定 2: observer 中心の image cell が変わった瞬間 (= primary cell 境界跨ぎ)
  const c0 = imageCell(p0, obs, L);
  const c1 = imageCell(p1, obs, L);
  return c0.kx !== c1.kx || c0.ky !== c1.ky;
}
```

OR 結合の双方向判定で漏れなし。判定 1 が defensive、判定 2 が primary。設計議論は本ファイル末尾の Appendix A。

## それぞれが解決するもの

1. **「-L で本当に +L に飛ぶ」 = 距離計算が最短画像で物理 PBC**
   - `pastLightConeIntersectionWorldLine` / `findLaserHitPosition` / `processHitDetection` 内部の `Δ = a - b` を `minImageDelta(Δ, L)` 経由に
   - レーザーが境界を跨いで反対側の相手に届く / 過去光円錐が反対側から回り込んで届く / hit 判定も最短画像距離

2. **「観測者が画面中央、他は最短画像位置に映る」 = `displayPos`**
   - 既存の `transformEventForDisplay` (or 同等の display 変換) に boundaryMode === "torus" のとき `displayPos` を組み込み
   - observer は常に画面中央、他オブジェクトは observer 中心の primary cell `[-L, L]²` 内に最短画像で配置 (Asteroids 風)

3. **「世界線が画面を横切らない」 = wrap 跨ぎ瞬間のみ線分 skip**
   - worldLine renderer 各種 (debris / dead / laser past-cone trail / 自機 / 他機) の LineSegments 構築箇所で `isWrapCrossing` を呼ぶ
   - 1 tick の物理進行は L よりずっと小さいので、判定 2 (cell 異なる) は wrap した瞬間 1 tick だけ true、その 1 tick 分だけ線分を skip

## レーザー軌跡 — (b) 案: 解析的に primary cell 境界で分割

レーザーは worldLine ではなく直線 `emissionPos + direction × t` で描画される。境界を跨いだ場合、`emission` から `tip` を直接結ぶと画面横切る → primary cell 境界との交点で **2 segment に分割** して描く。

### 実装

```ts
// laserPhysics.ts 等に新規 helper
function clipLaserToObserverCell(
  emission: {x, y},
  direction: {x, y},
  range: number,
  observer: {x, y},
  L: number,
): Array<{ start: {x, y}, end: {x, y} }> {
  // 1. emission を observer 中心 displayPos で primary cell 内に折り畳む
  // 2. observer の primary cell `[obs.x-L, obs.x+L) × [obs.y-L, obs.y+L)` を
  //    leaser 直線でクリップ (Liang-Barsky 風の sweep で `s_in`, `s_out` ∈ [0, range])
  // 3. cell 境界跨ぎがあれば、 出てから入るまでの間隙を image cell に飛ばして再開
  //    (observer の隣接 cell に入ったら s_in_next, s_out_next を計算)
  // 4. 結果は最大 N segment (通常 1 個、境界 1 跨ぎで 2 個、複数 cell 跨ぐ稀な高速 leaser で >2 個)
}
```

通常運転では `LASER_RANGE = 10` 程度なので、 1〜2 segment で済む (primary cell 全幅 2L = 40 に対し leaser が 10)。

### hit detection 側

`findLaserHitPosition` 内の「leaser と target worldLine の交差判定」も最短画像距離で。 worldLine の各点と leaser line の最近距離を `minImageDelta` で計算。

leaser 自体が長い (10 unit) ので、 1 つの leaser 内で leaser 始点と終点の最近 image が異なる可能性 (前回議論した worldLine の例と同じ)。 これは:
- 単純: leaser を観測者中心 primary cell + 隣接 image にクリップして得た segment 群について、 各 segment と worldLine の最近距離を取る
- または: leaser 直線を unwrapped 軌跡として持ち、 worldLine を 9 image 全部に複製して比較
- 前者の方が「クリッピング」と「hit 判定」を統一できて clean

### 漏れチェック

- emission が境界 (= 自機が wrap 跨ぎ瞬間に発射) に位置する場合: segment 1 が 0 長で segment 2 が反対側から始まる、 ただ視覚的には 1 segment と区別つかない。OK
- leaser が複数 cell を跨ぐ (例: leaser 始点 cell 0、 終点 cell 2 over wrap): 通常 LASER_RANGE 比 cell 全幅で起きないが、 念のため Liang-Barsky を ループで複数 cell 対応にする

### 隣接 cell image 描画 (= Asteroids 風 visual wrapping)

現状の設計は「観測者の primary cell `[obs-L, obs+L]²` 内のみ描画」。境界周辺に他機がいるとき、 反対端にも他機の image が見える Asteroids 演出は **今回はやらない**。

将来追加する場合:
- 各オブジェクトを 9 image cell `(kx, ky) ∈ {-1, 0, 1}²` に複製して描画
- imageCell 概念がそのまま再利用できる
- レーザーも (b) 案を (c) 案に置き換える形で 1〜4 image 描画

レンダリングコスト 9 倍に近いが、 PBC 演出としては完備感が増す。 Phase 7 として後追加可能。

### 過去光円錐の wrap 描画

`LightConeRenderer` は球面 mesh で描画。 観測者中心のため境界跨ぎ概念無し、 boundary mode に影響されない。 OK 変更不要。

### アリーナ ∩ 過去光円錐の交線 (`ARENA_PAST_CONE_OPACITY`)

現状 `ArenaRenderer` で円柱 ∩ 過去光円錐の交線を描いている (= 境界の事象)。 torus mode では「正方形枠 ∩ 過去光円錐」の交線が 4 平面 × 円錐 = 複雑な計算。 当面 torus mode では描画しない (= 開放問題)。 必要なら後で実装。

### Performance

- worldLine 描画で wrap 跨ぎ判定が 1 frame あたり N (= worldLine 点数) 回呼ばれる。 cost は floor + compare 2 軸、 negligible
- 距離計算 helper は既存の delta 計算を関数呼び出しに置き換えるだけ、 cost ほぼ同等

## Appendix A: wrap 跨ぎ判定の漏れ議論 (= 設計の検証)

### naive 判定 (`|wrappedΔ| > L`) の漏れ

broadcast 欠落で他機の phaseSpace が 1 sec 落ちた場合:
- 受信 worldLine = `[(10, 0), (32, 0)]` (= 22 unit ジャンプ、 間の点が無い)
- observer = (0, 0)、 各点の display = `(10, 0)` と `(-8, 0)`
- displayΔ = -18、 `|displayΔ| = 18 < L = 20` → 判定漏れ、 線繋ぐ
- 結果: 画面の `+10` から `-8` を結ぶ直線が描画される = 画面横切る

### 修正判定式

```
isWrapCrossing(p0, p1, obs, L) =
  (|p1.x - p0.x| > L) || (|p1.y - p0.y| > L)         // raw Δ defensive
  || (imageCell(p0, obs, L) ≠ imageCell(p1, obs, L)) // observer 中心 cell
```

### ケース別検証

| ケース | raw Δ | imageCell | 判定 |
|---|---|---|---|
| 通常 1 tick (Δ=0.02) | small | 同じ | 繋ぐ ✓ |
| 境界跨ぎ瞬間 (19.99 → 20.01、observer=0) | small | (0,0) → (1,0) | 切る ✓ |
| broadcast 欠落 (Δ=22) | >L | — | 切る ✓ (判定 1) |
| 1 周分 wrap (Δ=50) | >L | — | 切る ✓ |
| 観測者の真反対 (P_0=15, P_1=−15、observer=0) | >L (raw=30) | — | 切る ✓ |
| 観測者周辺の微振動 | small | 毎 tick 切替 | 毎 tick 切る ✓ (PBC として自然) |
| 観測者が wrap 跨ぐ瞬間 | — | 全点が同方向にスライド、 segment 内では cell 不変 | 繋がる ✓ |

判定 1 が異常検知 (defensive)、 判定 2 が正常な境界跨ぎ (primary)。 OR 結合で漏れなし。

## Appendix B: Step 2 — 3x3 image cell 描画 (= Asteroids 風 visual wrapping)

odakin 提案の「観測者中心に世界が無限に繰り返す」 visual の設計議論。 中間版完遂 + 実機 multi-tab 検証 + deploy 後の段階で着手予定。 実装着手時に別 plan ファイル (`plans/2026-04-XX-pbc-torus-tile-N.md`) を切ってから進める。

### 動機

中間版は「observer primary cell 内のみ描画」 で、 境界跨ぎ瞬間に world line が break して反対端から再開する Asteroids 挙動。 これは PBC の implementation detail を見せている状態 (= 「画面端で線が切れる」)。

「観測者中心に世界が `(2R+1)²` マス繰り返し」 は PBC topology の視覚化として本質的に正しい (= 「世界は周期的、 同じ自分が無限の image cell に居る」)。 相対論ゲームの教育的価値も高い。

### 核設計: InstancedMesh + offset attribute

各 D pattern renderer の geometry を InstancedMesh で `(2R+1)²` instance 化。 各 instance に offset attribute `(kx, ky)` を持たせて、 vertex shader で `transformed.xy += 2L * offset` で対応 image cell に shift。

```glsl
// shader
attribute vec2 aImageOffset; // (kx, ky) ∈ {-R, ..., R}²
// in begin_vertex 直後:
transformed.xy += 2.0 * uTorusHalfWidth * aImageOffset;
// fold は不要 (instance offset で既に正しい cell に置かれてる)
// timeFade / innerHide はそのまま動く
```

CPU 側は instance count = `(2R+1)²` で setMatrixAt しなくて良い (= 同じ instance matrix を共有、 differ は offset attribute のみ)。 instance attribute は 1 回 setup で固定 (= R 不変なら).

GPU cost ~9倍 (R=1) または ~25倍 (R=2)。 modern hardware で許容範囲。

### 副次設計判断

#### (a) timeFade に spatial fade 追加

現状 `fade = r²/(r² + dt²)` (dt only)。 隣接 image cell の vertex は dt 同じ (z 不変) → 同じ fade、 つまりメイン image と同強度で描画される。 自機本体が周囲 8 image に同じ強度で並ぶのは visual 過剰。

物理的整合 + visual 自然さのため:
```
fade = r² / (r² + dt² + s²)   ただし s = 観測者からの spatial 距離
```

s は modelMatrix 後の spatial xy magnitude (= `length(modelMatrix * transformed.xy)`)。 これで:
- 隣接 image cell の vertex は s ~ 2L で fade ≈ r²/(r² + dt² + 4L²) ≈ r²/4L² で大きく減衰
- 「無限平行世界」 が遠方で自然 fade out → 描画範囲 R を有限にしても破綻しない

ただし既存 timeFade 仕様変更で「等時刻面強調」 visual も変わる (= 別の設計判断、 odakin 確認)。

#### (b) innerHide の primary-only dispatch

自機 worldLine の hide center は player 現在位置 (world coords)。 隣接 image cell の vertex (= raw 値が `p + 2L * offset`) は hide center から world 距離 ~2L で hide されない (= 描画される)。 つまり 3x3 で描画すると **「自機本体が周囲 8 image cell に並ぶ」** 状態になる。

これは Asteroids 的に自然 (= 自機が境界に近づくと反対側にも自分が出る) だが、 自機本体 hull が 9 体並ぶと visual 過剰。 hide center を 9 image cell に複製するか、 自機本体 mesh は primary image (= offset = (0, 0)) だけ描画する dispatch が要る。

選択:
- **(b-1) hide center 複製**: `uInnerHideCenter` を 9 個 uniform array に → vertex shader で 9 つの距離計算 → 最小値で hide 判定。 cost 9 倍だが visual は完全 (= 各 image の自機本体が hide される)
- **(b-2) primary-only dispatch**: 自機本体 SelfShipRenderer のみ instance count = 1 (offset = (0, 0) のみ)、 worldLine / debris / laser は 9 instance。 visual: 自機本体は primary image だけ、 周囲 image には worldLine の延長や laser のみ
- **(b-3) 何もしない**: 自機本体が 9 体並ぶ visual を許容。 「無限平行世界」 として自然 (= 各 image cell に「自分」 が居る)

odakin の好みで判断。 実装難易度は (b-2) < (b-3) < (b-1)。

#### (c) 視野半径 R の constants 化

`ARENA_TILE_RADIUS = 1` (= 3x3) を constants 化。 R=2 (= 5x5) に切り替え可能に。 視野 viewport から動的算出 (= camera frustum と arena cell サイズから R を計算) は Phase 4 として別タスク。

### 影響範囲

全 D pattern renderer に instance 化が要る。 各 renderer の現状:

| Renderer | 現状 | 変更 |
|---|---|---|
| WorldLineRenderer | 単一 mesh × 複数 segment | InstancedMesh × 複数 segment、 instance count = `(2R+1)²` |
| LaserBatchRenderer | 単一 LineSegments | InstancedLineSegments? (= three.js 標準なし、 自前 instancing or 9× vertex 化) |
| DebrisRenderer | 既に InstancedMesh | instance count を ×9、 offset attribute 追加 |
| SelfShipRenderer | 単一 mesh hierarchy | (b-2) なら instance 1 のまま、 (b-3) なら InstancedMesh × 9 |
| OtherShipRenderer / RocketShipRenderer / JellyfishShipRenderer | 単一 mesh hierarchy | InstancedMesh × 9 |
| LighthouseRenderer | 単一 mesh | InstancedMesh × 9 |
| DeathMarker / 過去光円錐 etc | 単一 mesh | InstancedMesh × 9 |

LineSegments の instance 化は three.js native でサポートされない (= LineSegmentsGeometry / Line2 等の addon が要)、 vertex を 9 倍してマニュアル offset 付与の方が simple。 worldLine は TubeGeometry (= mesh) なので InstancedMesh で OK。

### Phase 分け

- **Phase 1 (prototype)**: worldLine だけ 3x3 描画 (InstancedMesh 化 + offset attribute + shader 加算)。 visual evaluation → 採用判断
- **Phase 2 (採用なら、 base infrastructure)**: torusFoldShader を `torusTileShader` に拡張 (offset 付与 + 既存 fold は撤去)。 timeFade に spatial fade 追加判断 → 別 shader (`spatialFadeShader`) で導入
- **Phase 3 (展開)**: 全 D pattern renderer に instance 化。 SelfShipRenderer は (b) のうち選択した dispatch を実装
- **Phase 4 (任意)**: shader 最適化 (= occlusion culling、 R を viewport から動的算出)

### 開放問題

- **過去光円錐 ∩ 全 image cell の交線**: 各 image cell に独立な arena 枠が描画される? primary cell のみ? 設計判断
- **Radar 表示**: 隣接 image の他機をどう表示する? 重複表示するか、 primary のみ表示するか
- **死亡 routing / hit 判定**: 中間版で最短画像距離で正しく動く。 描画上の隣接 image vertex は「visual 装飾」 のみで物理判定には影響しない (= 中間版の挙動を維持)
