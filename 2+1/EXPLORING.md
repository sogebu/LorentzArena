## スマホ UI の設計思考（2026-04-06、未決定・未着手）

**Status**: 未決定・実装着手前の option space 分析。実装判断の前に再評価すること。

### なぜこれを考えたか

現状 2+1 の操作系は PC キーボード前提（W/S 加速、矢印キーでカメラ、Space レーザー）でスマホでは完全に遊べない。GitHub Pages で公開している以上、スマホでアクセスしてくる訪問者は高確率で存在し、**3 秒で離脱される**。物理デモとしての到達範囲と、新規ユーザー獲得の両方に直結する。

### 再枠組み: スマホは「別ゲーム」として設計する

PC の操作セットを touch に map する発想を捨てる。

- W/S 同時押しで「加速しながら方向転換」は touch で 2 本指必要
- Space 発射は PC では反応速度の快感だが、touch ではレイテンシ + 誤タップで別物
- 矢印キーでカメラ pitch は PC の余韻で、**2+1 ゲームでは pitch は純粋に観賞用**（movement は x-y 平面）

代わりに「**物理デモとしての不可侵要素を特定 → それを最小の touch 語彙で表現**」の順で設計する。

### 物理デモとしての不可侵要素

| 現象 | 体感条件 |
|---|---|
| 時間の遅れ | **持続的に高速（>0.9c）で動く** ← 加速を 5〜10 秒維持する必要 |
| ローレンツ収縮 | 同上 + 他プレイヤーの世界線を見る |
| 因果律 | レーザーが過去光円錐到達時に発火する kill 演出 |
| 過去光円錐 | 他プレイヤーの worldline が visible になる条件 |

**帰結**: 持続的加速が死守ポイント。**スワイプ式のパルス操作やタップ移動は、この 1 点で失格**。加速が断続的になると c に近づけず、時間の遅れが体感できない。

### 操作の棚卸し（PC → mobile）

| PC 操作 | 役割 | mobile 必要度 | 候補マッピング |
|---|---|---|---|
| W/S | 前後スラスト（heading 方向） | **必須** | 仮想 stick radius / 上下ドラッグ |
| 矢印 yaw | 向き変更 | **必須** | 仮想 stick angle / 左右ドラッグ |
| 矢印 pitch | カメラ tilt | 不要 | 固定 or ピンチ |
| Space | 発射 | **必須だが自動化可** | タップ / 自動発射 |
| 正射影切替 | 視点種別 | あれば嬉しい | menu |
| 静止系切替 | 視点フレーム | あれば嬉しい | menu |

**必須 3 つ** = heading + thrust + fire。

### 候補スキーム

| スキーム | heading | thrust | fire | 視認性 | 物理体感 | 疲労度 |
|---|---|---|---|---|---|---|
| **A. Dual stick** | 左 stick angle | 左 stick radius | 右 stick / 右タップ | **大**（2 隅） | ◯ | 中 |
| **B. Single stick + tap fire** | 左 stick angle | 左 stick radius | 画面上半分タップ | 中 | ◯ | 中 |
| **C. Single stick + auto-fire** | 左 stick angle | 左 stick radius | 自動（cooldown 毎） | **小**（1 隅） | ◯ | **小** |
| D. Tilt + tap | デバイス傾き | 前傾 | タップ | 小 | △ 傾け維持きつい | 大 |
| E. Tactical lock-on | 敵タップ追尾 | 自動 | 自動 | 小 | ✕ 自分で加速しない | 小 |

**D 脱落**: 持続加速と傾け維持が矛盾。机置き使用も壊す。
**E 脱落**: 自分で加速しないと時間の遅れが体感不能、物理デモの核が死ぬ。

**残る候補: A / B / C**。視認性・疲労度で **C が有利**。

### 深く考えるべき論点 3 つ

#### (1) fire ボタン問題 → auto-fire は物理デモとして正しい

PC の Space は「反応速度で勝負する射撃」の快感を生むが、touch で同等反応は出ない。auto-fire にすると **ゲームが「射撃の上手さ」から「位置取りの上手さ」にシフト**する。

これは一見 gameplay の喪失だが、**物理デモとしてはむしろ純化**する。位置取り = 世界線の引き方 = 相対論の練習。auto-fire なら「敵の過去光円錐に入る位置に自分の未来光円錐を置けるか」という教育的ゲームになる。

→ **auto-fire は物理デモとして積極的に正しい**。PC 版にも逆輸入検討の余地あり（別議論）。

#### (2) 持続加速の指疲労問題 → cruise control

Single stick を 10 秒間フルで倒しっぱなしは物理的に辛い。でも c に近づくには 5〜10 秒の持続加速が必須。矛盾する。

**解決案**: joystick をダブルタップ or 長押しで「現在の angle + thrust にロック」
- ロック中は視覚的に stick が固定表示（発光など）
- どこかをタップで解除
- ロック中も angle だけドラッグ微調整可能

副次効果: **「慣性航行（free fall）」を自然に体験できる**。ロックしたまま手を離して時計を見つめる使い方が成立する（物理的に意味がある）。

#### (3) 視認性 vs 操作面積のトレードオフは 3 方向で解ける

1. **最小化**: 操作面積を極小化（C + cruise control）
2. **画面外化**: edge swipe 等で画面に描画しない（PWA standalone mode 必須 → Web では非現実的）
3. **美学化**: 操作 UI を physics の visual 語彙で描く
   - joystick を「自分の reference frame の基底ベクトル」として描画
   - stick を倒すと「自分の 4元速度ベクトル」のプレビューが時空図に出る
   - **操作 UI 自体が物理の教材になる**

**方向 3 は創造的に魅力的**。スマホ版独自価値になりうる。実装コストは未知。

### 縦画面と横画面の役割分担

両方に同じ機能を載せる必要はない。役割を分けるほうが誠実:

**横画面（本番ゲームプレイ）**
- 親指 2 本保持、左下に joystick
- **候補 C（single stick + auto-fire）+ cruise control**
- 画面上端に薄く HUD（固有時刻・世界時刻）
- 右上に小さな menu ボタン（projection / frame toggle 収納）

**縦画面（one-handed / 観賞 / オンボーディング）**
- **観賞モードだけ実装、簡易プレイは諦めて横持ちに誘導** が潔い
- 上 100% = 世界（自動回転カメラ）、操作なし
- 「自分は参加していない、世界の時空図を眺めている」
- コーヒーブレイクで開く用途、「何だこれ？」の第一印象を掴む
- 「横向きにすると参加できます」というヒント

理由: 縦で遊べるゲームにしようとすると UI 領域が広がり、視認性を毀損する。縦は「観賞」「入り口」「物理デモのティーザー」に特化。

### オンボーディング問題

PC ユーザーは「W を押したら動いた、Space を押したら何か出た」と自力学習できるが、**スマホユーザーは 3 秒で離脱**する。最初の 3 秒で何かが起こらなければならない。

候補:
- 初回起動時に joystick を **幽霊のように自動で動かす 2 秒アニメ** → 「これを触る」と分かる
- 最初の 5 秒、画面中央に **「固有時刻: X / 世界時刻: Y」を巨大表示** → 「時間が 2 種類ある」ことを即座に示す
- 他プレイヤーが画面内にいる場合、その世界線を強調描画 → 「他人と自分の時空が違う」ことを示す

**スマホ版の成否を決める**ので深く考える価値がある。

### 暫定の有力候補（2 つ、まだ決定しない）

#### α. 保守的ミニマム
- 横画面: single stick + auto-fire + cruise control
- 縦画面: 観賞モードのみ
- 操作 UI は普通の半透明 joystick
- menu に projection / frame toggle
- オンボーディング: joystick 幽霊アニメ + 固有時刻の巨大表示

**強み**: 実装が素直、確実に動く、物理デモ価値 preserved
**弱み**: スマホ版独自性なし

#### β. 創造的
- 横画面: 同上
- **joystick を「4-velocity の空間成分ベクトル」として描画**、倒すとその先に自分の未来光円錐プレビュー
- cruise control ロック時は joystick が発光する相対論的ベクトルのまま固定
- 縦画面: 観賞モード + 簡易説明パネル
- オンボーディング: joystick = 4-velocity の物理的意味を最初の 10 秒で説明

**強み**: スマホ版が物理デモとしての独自価値を獲得。「スマホを持って相対論ベクトルを触る」体験
**弱み**: 実装コスト未知、デザイン難度高、外すとダサい

**α から入って β の方向に育てるのが現実的**。ただし実装着手前に再評価すること（この段階では仮）。

### 2026-04-10 の設計議論と方針決定

#### 暫定決定事項（DESIGN.md promote 前の合意）

**大方針: α（保守的ミニマム）で入る。** β（joystick = 4-velocity）は α 実装後に育てる方向。

**操作スキーム: 全画面タッチジェスチャ（UI 要素ゼロ）**

議論の過程で候補 A〜E → ボタン+スティック → レバー+ジェスチャ → ボタン+ジェスチャ → 全画面ジェスチャへ収束。描画 UI 要素をゼロにし、画面 100% を 3D ビューに使う。

画面全体が見えないジョイスティック。タッチ開始点を基準に、そこからの変位で操作。

| 操作 | 入力 | 備考 |
|---|---|---|
| heading (yaw) | 横スワイプ | カメラは heading に自動追従（PC 版と同じ） |
| thrust (加速/後退) | 縦スワイプ | 上=前進加速、下=後退。タッチ開始点が基準、変位量で推力の強さが変わる |
| fire (発射) | ダブルタップ（2 回目を保持） | 保持中 cooldown ごとに連射 |
| heading + thrust | 斜めスワイプ | 方向転換と加速が同時にかかる |
| fire + heading + thrust | ダブルタップ後にスワイプ | 全操作を 1 本の指で同時実行 |

```
┌────────────────────────────────────┐
│                                    │
│       ← heading →                  │
│       ↑ 加速                       │
│       ↓ 後退                       │
│       ダブルタップ → 射撃          │
│                                    │
│       （画面全体、UI 要素ゼロ）    │
└────────────────────────────────────┘
```

**描画 UI 要素ゼロ。** 画面のほぼ全体が 3D ビュー。HUD（速度計・固有時刻等）は薄く重ねて表示。

**設計根拠と議論の経緯**:

1. **PC 操作を touch に map する発想を捨てる** — EXPLORING.md 原案の核心はそのまま維持
2. **全画面が 2D コントロール面** — タッチ開始点を原点に、X 軸=heading、Y 軸=thrust。固定位置のジョイスティックと違い、どこを触っても基準点がリセットされるので指疲労が軽い
3. **thrust が連続値になる** — 変位量で推力の強さが変わるため、中間速度巡航が自然に実現。加速ボタン（binary）やレバーの検討課題が解消される。0.3c と 0.9c の世界の違いを意図的に比較できる（物理デモ価値）
4. **後退も自然に統合** — 下スワイプで heading 逆方向への推力。S ボタンの検討課題も解消
5. **pitch 省略** — 2+1 では movement は x-y 平面。カメラ pitch は観賞用だが優先度低い
6. **ダブルタップで射撃** — シングルタッチ=heading+thrust、ダブルタップ=射撃。2 回目のタップを保持したまま指を動かせば全操作の同時実行が成立
7. **auto-fire 不採用** — 常時発射は画面が busy になる。ダブルタップなら射撃タイミングの判断が残り、ゲーム性を保てる
8. **ダブルタップ検出の遅延** — 最初のタップ後に ~200ms の待ち（次のタップが来るか判定）が入り、heading/thrust スワイプの開始が少し遅れる可能性。ゲーム用途では許容範囲と判断するが、実機テストで要確認
9. **指を離すと推力ゼロ** — 摩擦で自然に減速。別の場所をタッチすれば新しい基準点で操作再開

### open questions（実装前に要検証 / 要決定）

- **WebRTC on mobile Safari の信頼性**: 実機テスト未済。単独動作（ソロ観賞）を担保するフォールバックが必要かもしれない
- **ダブルタップ検出の閾値**: タップ間隔（~200-300ms）と位置許容範囲の調整。速いゲームプレイで誤検出/見逃しの頻度は実機で要確認
- **縦スワイプの thrust 感度**: 変位量→推力のマッピング（線形 vs 非線形）。小さい変位で微調整、大きい変位で全力加速、のカーブが物理デモ体験に影響
- **世界線描画の視認性**: スマホの小さな画面で world line の TubeGeometry がどこまで読めるか。LOD 調整が必要かも
- **画面の向き変更ハンドリング**: 縦横切替時に観賞 ⇔ プレイが滑らかに遷移すべきか、リスポーンが要るか
- **バッテリー消費 / 発熱**: three.js + R3F + 120Hz 物理ループは mobile で非現実的な可能性。`requestAnimationFrame` + `setInterval` の共存が熱問題にならないか
- **「用語再考」との合流**: スマホ版リリースを契機に KILL → INTERCEPT 等の語彙変更も一気にやるべきか、別にやるべきか（本ファイルの「用語の再考」セクション参照）
- **iPhone のホームインジケータ / 安全領域**: 画面下端のスワイプが iOS ジェスチャ（ホーム・コントロールセンター）と衝突しないか

## 用語の再考（未決定・言い換え候補）

**Status**: 2026-04-06 に DESIGN.md に記録、2026-04-07 に 4 軸レビューで pure exploration として EXPLORING.md に migrate（元々は DESIGN.md に `### 用語の再考` ヘッダー付きで追加されたが、同日 `88ed267` コミットで「残存する設計臭」追加時にヘッダーが置換され orphan 化していた）。**未決定・未適用**。コード・UI・ドキュメントの全域に現在も戦闘系語彙（KILL / DEAD / 死亡 / 撃破 / kill / deathFlash 等）が残存。置換する場合は機能的整合を崩さないように一括リネームで別コミット予定。

### Why 再考

本アプリの本質は「相対論の時空図可視化」で、物理デモ/教育用途に寄せるなら戦闘系語彙は重い。現在の社会的文脈でも、`KILL` を画面に出すマルチプレイヤー Web アプリは不必要に不穏。

### 却下した方向（ぬるい言い換え）

「浄化」「ラヴ」「LOVE」など過度に婉曲な語。Euphemism treadmill で却って不気味さが増す（ゲーム業界で知られた現象; Portal の GLaDOS 的な皮肉に読まれる）。ネガティブな意味が言い換え先に染みて結局同じ、あるいは作者の意図を疑われる逆効果。**やばい**。

### 有力候補（物理記述寄り）

「そもそも物理的に何が起きているか」を中立に記述する方向。インターセプト（迎撃）/ コンタクト（接触）/ リゾルブ（解決）/ デカップル（切り離し）/ オフライン / ポーズ など。

## 過去光円錐交差点エフェクトへの遠近法適用（未決定・未着手）

**Status**: 2026-04-11 に検討課題として登録。未設計・未実装。

### なぜ必要か

正射影モードでは、過去光円錐と世界線の交差点マーカーが視点からの距離に関係なく同じ大きさで描画される。交差点が光円錐の手前側にあるのか奥側にあるのか判別しづらく、他プレイヤーの世界線が観測可能（光円錐内）かどうかが分かりにくい。

透視投影モードでは three.js の投影行列が自然に遠近を反映するためこの問題は起きない。

### 案

- **正射影モードで交差点マーカーに手動遠近法を適用**: カメラからの距離に応じてマーカーの scale を調整し、手前は大きく・奥は小さく描画する
- 光円錐面を通過する世界線の太さ・opacity を段階的に変化させる（光円錐外=薄い、光円錐内=太い）

### 検討ポイント

- 正射影モード固有の問題（透視投影では自然に解決される）
- パフォーマンスへの影響（世界線のセグメントごとに光円錐内外判定が必要）
- 静止系表示と世界系表示で同じ手法が使えるか

## 進行方向・向きの認知支援（2026-04-15、育成パス Step 1 実装済）

**Status**: 13 案 (+追加 3 案) を 5 カテゴリに整理、優先順位の所感を記録。**Step 1 ((3) exhaust の visual-only 版) は 2026-04-17 に実装・デプロイ済** (commit `cdcad32`、build `2026/04/17 16:35:42 JST`)。Step 2・Step 3 と他機対応 (共変 α^μ broadcast) は未着手。実装詳細は DESIGN.md §描画「Exhaust」。

### なぜこれを考えたか

相対論に慣れていない人類は「自分がどっち向いて進んでいるか」を見失う。特に自分の静止系表示では、加速するとその方向に世界が遠ざかっていく (風景が流れてくる) ため「自分が進んでいる」感覚が逆転する。加えて `heading` (入力が加速を与える方向) と `velocity` (現在の 3 速度方向) が一致しないので、「向き」という概念自体が二重化している。

SESSION.md の TODO「自機および敵機のマーカーに向き (heading) を入れて…」は本枠組みの最小一案。本ファイルでは option space 全体を列挙。

### A. プレイヤーマーカーに情報を足す

1. **heading 矢印**: 入力で制御される方位を、自機球に付随する三角形/コーンで示す。(SESSION.md TODO)
2. **velocity 矢印**: 3 速度ベクトル (γv の空間成分) を別矢印で。heading と分離すれば「向いている」≠「流れている」を可視化
3. **推進ジェット (exhaust)**: 加速キー押下中、反対方向に尾翼/噴射粒子。入力→結果の feedback

### B. 固定位置 HUD 系

4. **クロスヘア**: カメラ中央に heading 射影 (照準も兼ねる)
5. **姿勢計/コンパス**: HUD 隅に circular 計。heading / velocity / 世界系 +x 軸を並置
6. **ミニマップ (世界系)**: 左下に縮小 world-frame xy ビュー。rest-frame 表示と独立

### C. 世界の基準を足す

7. **world-frame 軸ガイド**: 自機足元に +x/+y 軸 (赤/緑) を常時描画。フレーム切替時の anchor
8. **グリッド床**: xy 平面に薄いグリッド。静止系では均一、動けば Lorentz 収縮で歪む
9. **遠方の "星" 点群**: static 粒子群。光行差で速度方向前方に集まる → 運動方向のランドマーク

### D. 入力→結果の即時 feedback

10. **キー表示**: 自機周囲に W/S/A/D 矢印オーバーレイ、押下中ハイライト (初見プレイヤー学習支援)
11. **加速ベクトル線**: 加速キー押下中、加速の 3D 方向を半透明線で自機から伸ばす (velocity と別色)

### E. 履歴を使う

12. **モーション・トレイル**: 世界線描画で最近の数秒だけ明るく (速度方向も視覚化)
13. **シャドウ・ゴースト**: 1 秒前の自機位置にうっすら残像 (動きの速度感)

### 優先順位の所感 (対戦実用性 × 実装負荷)

| 順 | 案 | 理由 |
|---|---|---|
| 1 | 推進ジェット (3) | 入力の視覚的 feedback が最も直感的。heading/velocity の概念を事前に理解させずに済む |
| 2 | velocity 矢印 (2) | 相対論学習の要。heading との食い違いが目で分かる |
| 3 | heading 矢印 (1) | 既に TODO 入り、実装軽い |

| 5 | ミニマップ (6) | rest frame 切替時の混乱を根本解消、HUD 面積コストと引き換え |

その他 (キー表示、コンパス、トレイル) はチュートリアル段階で効きそう。

### 相互作用と排他

- (1) + (2) + (11) は同時に使うと「heading / velocity / acceleration」の 3 ベクトルを分離可視化できるが、矢印が多すぎると読めない。色分けルール (heading=白、velocity=自機色、acceleration=半透明ジェット等) が必要
